# 课程蓝图：一种追本溯源的教学哲学

如果你问一个老师"这节课教什么"，他会给你一份大纲。但如果你问"为什么要这样教"，大多数人的回答会含糊其辞。这两者之间的差距，就是"课程蓝图"要填补的东西。

这个词源于我和同事的一次日常对话。每次上课前，梁嘉伟都会来问我同样的问题："这节课的核心思路在哪？"我跟他聊完，他就能做出一版效果很理想的内容。久而久之我们意识到，这个在课前被传递的东西，需要一个名字——课程蓝图。

它不是课程大纲，不是教案，也不是知识点的排列组合。它是隐藏在这一切背后的那个**核心思路**——讲这门课的人到底是怎么理解这个领域的，他选择这条路径而不是那条路径的根本原因是什么。

## 教 Git 的故事

要理解课程蓝图是什么，最好的方式是用一个例子。

大多数人教 Git 是这样教的：先讲版本控制的概念，然后列出几个常用命令，让学生跟着敲一遍。这种教法的问题在于，学生在学完之后只知道"怎么用"，但不知道"为什么这样用"。

我的教法是反过来的。我会先花大量时间讲：Git 是怎么被发明出来的？发明它的时候，Linus Torvalds 假设的场景是什么？当时的替代方案 SVN 是什么样子的？中心化和分布式到底有什么区别？

把这些讲清楚之后，再反过来讲暂存区、工作区、add、commit、push。你会发现，之前那些让新手困惑的概念——为什么需要暂存区？为什么提交到本地之后还要推送到远端？——全都变得顺理成章了。

这就是课程蓝图的运作方式：**不是从结论出发，而是从源头出发。**

## 追本溯源

"溯源"是课程蓝图最核心的特征。

这不是一种教学方法，而是一种教学哲学。它的基本信念是：任何一个知识、工具或方法，都不是凭空产生的。它们诞生于某个具体的场景，为了解决某个具体的问题，经历了某种特定的演化路径。如果我们能回到那个原始场景，理解那个问题，走一遍那条演化路径，那么所有的"知识点"就不再是需要死记硬背的碎片，而是逻辑链条上的必然环节。

这个信念不仅在教别人的东西时成立，在教我们自己创造的东西时也成立。

比如量潮数据这个产品，它的诞生纯粹是因为我们自己搞科研的时候有了需求，然后周围的人也有同样的需求，找上门来，我们干脆就把这件事商业化了。这个起源故事本身就包含了对产品本质的深刻理解。

再比如我们的周会制度。当初做改革的时候，我没有直接找管理模板，而是去翻各个国家的政治史，研究议会是怎么演化的。现在的这套动态调整模式，其底层逻辑就来源于以英国历史为主线的政治演化——这是一种留出了动态调整空间的结构，而不是一个固定的流程模板。

## 如何制作课程蓝图

如果你想做出符合这种风格的课程蓝图，核心就两件事。

第一，**了解它的发展历史**。这个东西是怎么诞生的？诞生时面对的是什么问题？它经历过哪些关键的演化？这些演化的驱动力是什么？

第二，**从里面提炼出关键点**。历史是具体的，关键点是抽象的。读完了 Git 的发展史之后，提炼出"分布式 vs 中心化"这个核心对比；研究完了议会史之后，提炼出"留出动态调整空间"这个结构原则。这个提炼的过程是整件事里最难的一步，但也是最关键的一步。

提炼完之后呢？把它放到真实场景里去用。这不是一个线性结束的过程——你会不断地在新的场景中发现新的问题，然后回去重新溯源、重新提炼。

## 为什么重要

课程蓝图之所以值得被单独拎出来说，是因为它直接决定了我们的课程体系跟市场上大多数课程的根本区别。

大多数课程的设计思路是：这门课要覆盖哪些知识点？按什么顺序讲？这种思路天然倾向于往课程里塞"干货"，越多越好。但干货的堆积不等于有效的教学——用户学到的东西是一堆没有连接起来的孤岛。

我们的思路是相反的：这门课要解决的核心问题是什么？它的源头在哪里？这种设计思路天然倾向于"少而深"——深度理解一个核心原理，比泛泛了解十个表面概念有用得多。

这就是我们经常说的"道先于术"。"道"不是玄学，它就是那个被你从历史中提炼出来的核心思维模型。学会了"道"之后，学生能够自己把这个思维方式用到各种不同的问题上去，而不需要老师提前把所有场景都预判到。

这也是为什么我们的课程天然擅长小而精的模式——当我们面对一个特定群体的时候，我们会本能地进入他们的情境，去理解他们的背景和需求，然后从那个情境出发去设计课程。这种能力高度依赖对用户的深入了解，规模化的瓶颈很明显，但它带来的好处是：客户的粘性极高，我们的课程在真正理解它的人眼里是无价的。

## 不是终点

课程蓝图不是一个可以一次做完然后反复使用的东西。它需要不断地被审视、被追问、被重新理解。

每当你面对一个新的教学场景，发现原来的蓝图解释不了的时候，就是该回去重新溯源的时候了。真正有效的课程蓝图，不是来自精心设计的教案，而是来自不断追问"为什么"的习惯——**先搞清楚为什么这样教，再想清楚教什么，最后才是怎么教。**

这个顺序不能乱。
